```ts
  diet?: 'vegan'|'gluten-free'|'halal'|null;  // [Волна 5–6, G1/G3] пред-фильтр диеты из deep-link (?diet=vegan → /menu/vegan, /plan/helper). При заданном — шаг 3 (своё меню) авто-фильтрует на dietBadges, как детское на childFriendly.
  childGuests?: number;        // [Волна 1, B-OLGA-5] число детей (детский прайс по CHILD_RATE)
  guestEdge: 'none'|'over-500';  // over-500 → рендер CTA «Оставить заявку», НЕ slider
  tierMode: 'preset'|'custom';
  tier: 'economy'|'standard'|'premium'|'luxury'|null;
  selectedItems: SelectedItem[];      // custom-режим (B-MUX-3/П1): { pricePerGuest; qty(=порций на гостя); station; coversGuests; excludeAllergen?; ... }
  liveSummary: { guests; format; tier; perGuest; base; addonsTotal; total; savings; service; serviceBreakdown? }; // B-MUX-1/П5: единый running-итог, пересчёт через calcTotal <50ms, count-up. [Итерация 4, SERVICE] service+serviceBreakdown тянутся из calcTotal (строка «Сервис-норма», биндинг SERVICE_DELIVERY_SPEC §1/§2/§4). ИМЕНА полей — ровно как return calcTotal (`07`): base / addonsTotal / total / service / serviceBreakdown (Итерация 5, единый source of truth имён, закрывает рассинхрон subtotal/addOnsTotal).
  addOns: AddOn[];                    // БЕРЁТСЯ из единого ADDONS (07), НЕ свой список (B3/КР-3)
  contact: { name; phone; date; comment };
  base; addonsTotal; total; perGuest; savings; service; serviceBreakdown?; // [Итерация 4, SERVICE] service — прозрачная расшифровка «за что платим» (персонал + координатор + сетап + холодовая цепь), уже заложен в pricePerGuest (см. 04:1067, 07 calcTotal). ИМЕНА — как return calcTotal (`07`): `base` (НЕ subtotal) + `addonsTotal` (НЕ addOnsTotal) + `total` + `service`. Единый source of truth имён между 07↔08 (Итерация 5).

```
Persistence: **sessionStorage** (восстановление при refresh, очистка при success).

## Расчёт (из §15.2, канон цен — меню + единая матфункция)
- **ЕДИНАЯ `calcTotal(guests, format, tier, addons, opts)`** из `07` (см. секцию «Единая матфункция calcTotal») — импорт из `lib/pricing.ts`. Оба инструмента считают ОДИНАКОВО (устраняет КР-2).
- Preset: `const r = calcTotal(guests, format, tier, addOns, { discounts:true, bookingDays })`.
- Custom: `const r = calcTotal(guests, format, 'custom', addOns, { discounts:true, bookingDays, items: selectedItems.map(i => ({ pricePerGuest:i.pricePerGuest, qty:i.qty })) })`; мин. число позиций по формату/тарифу (как в 04).
  - **`qty` обязателен** (B7/КР-5): `item.total = i.pricePerGuest * i.qty * guests`. Без `qty` блюдо не считается.
  - **Цены тарифов (`PRICE_PER_GUEST`) — 🟡 `pending-verification`:** значения взяты из канона `NILOV_UNIFIED_MENU` (07),
  но **НЕ совпадают с прайсом на живом сайте NiloV** (Фуршет+Кофе-брейк от 390 ₽, Корпоратив от 2 450 ₽,
  Банкет/Свадьба от 4 470 ₽). Сверить с актуальным прайсом заказчика перед продом и, при необходимости,
  поменять канон меню в `07` (см. BUG-F2 отчёта FactChecker).
- Chef-at-home: `chefTotal = hourlyRate * hours + Σ(addon.perGuest*guests) + addonsFixed`. `hourlyRate` — **`pending-verification`** (см. `07` §Edge cases), НО конструктор НЕ уходит в тупик: показывает **базовую почасовую оценку «от 2 500 ₽/час (уточняется)»** (из `07`) + примерную сумму от `guestCount`×`hours`×`hourlyRateBase`, с пометкой «точная цена после брифа менеджера» (B-MUX-6). Если клиент хочет точный расчёт сразу — ветка **«Свяжитесь с менеджером»** → `/api/contact`, которая НЕСЁТ контекст (формат/гостей/дата/выбранные add-ons из State, не «позвоните» без данных).
- Dirty-check: «Вы экономите X vs Премиум» badge — `savings` считается от верхнего публичного тарифа (для банкета база Премиум 6 970, для фуршета/кофе-брейк — premium 5 350 / 2 450). **[Волна 7, A3] Устранён рассинхрон терминов:** вместо «vs Люкс» (internal-only `luxury`) бейдж НЕСЁТ публичный лейбл «Премиум» — совпадает с UI тарифа (см. `07` TIER_LABEL / `04` PackageGrid). Термин «Люкс» НЕ появляется у клиента ни в бейдже, ни в UI; если заказчик позже введёт публичный «Люкс/Гранд/Элит» — меняется ТОЛЬКО `TIER_LABEL`, бейдж подхватит автоматически.

## Live-итог на КАЖДОМ шаге (B-MUX-1 / П5)
Критик (`25_MENU_UX_CRITIC`) указал: итог виден только на шаге 5, 4 шага проходят «вслепую». **Решение:** `StepProgress` (сверху) и каждый шаг несут компактный **running-итог `liveSummary`**, пересчитываемый через ту же `calcTotal` (<50ms, count-up):
> «N гостей · Формат · Тариф · ≈X ₽/гость · Итого Y ₽» (+ бейдж «Экономите Z» при отклонении от Люкс).
- Шаг 1 (Формат): как только выбран формат — итог считается от `MIN_GUESTS[format]`, обновляется при смене.
- Шаг 2 (Гости): ползунок/цифра → итог live (count-up) при каждом изменении; quick-кнопки [20][50][100][150][300] — тоже.
- Шаг 3 (Своё): sticky-саммери = тот же `liveSummary` (устраняет дублирование «где моя цена»).
- Шаг 4 (Доп. услуги): добавление/снятие add-on → итог пересчитывается мгновенно (B3/КР-3).
- Шаги 5–6: тот же `liveSummary`, финализирующий вид.
Единый источник — поле `liveSummary` в §State, НЕ отдельный расчёт.

## Pre-select формата из deep-link (Волна 3, закрывает D8 — Дарья)
Критик (Дарья, D8): размер-подсказка появлялась только на шаге 2 — лишний экран для известного сценария (кофе-брейк, 15). **Решение:** при заходе по deep-link с параметрами (`?format=coffee-break&guests=15&tier=standard`, как пресет «Кофе-брейк на 15» на Hero/блок 1, D2) — **шаг 1 (Формат) пропускается**, рекомендованный формат (`coffee-break`) **подсвечивается сразу** (`selected` + gold border + check, Framer `layoutId`) ещё ДО любого экрана выбора; `guestCount` и `tier` предзаполняются из URL. Никакого промежуточного экрана — пользователь попадает сразу на шаг 3 (Тариф/Своё) с готовым контекстом кофе-брейка на 15 гостей. То же поведение для пресета «Свадьба на 80» (`?event=svadba&guests=80&format=banket&tier=standard`).
- `useConstructor` читает `searchParams` при инициализации (`format`/`guestCount`/`tier` — если валидны по `MIN_GUESTS`), ставит `currentStep` соответственно (пропуск шага 1 при наличии `format`).
- Подсветка рекомендованного формата НЕ блокирует: если `guests` меняется вручную — подсветка пересчитывается (см. B-MUX-5).
Критик: формат выбирается ДО гостей → size-подсказка не работает. **Решение (формат остаётся шагом 1 для нарратива, но становится рекомендательным):**
- На шаге 2 (Гости) сразу после ввода `guestCount` конструктор ПОДСВЕЧИВАЕТ рекомендованный формат (в шаге 1 и в `liveSummary`) по единому правилу `MIN_GUESTS`+размер:
  - **10–19 гостей** → «Для камерной группы — **Кофе-брейк** (от 10) или **Частное меню / Chef-at-home**» (фуршет/банкет недоступны по меню — инлайн-нотис из `07`).
  - **20–49 гостей** → «Рекомендуем **Фуршет** (мин. 20)».
  - **50+ гостей** → «Рекомендуем **Банкет** (посадка, официанты)».
- Подсветка — не блокировка: клиент может выбрать любой доступный формат, но рекомендация видна ДО финала (П2 «по размеру» реализован). Связь с `MIN_GUESTS` (`07` §MIN_GUESTS).

## Build-your-own порционно на гостя (B-MUX-3 / П1)
Критик: `qty` есть математически, но клиент НЕ собирает «свою линию» осознанно. **Решение — режим сборки «линии» по станциям/категориям:**
- `CustomMenuBuilder` группирует каталог по станциям (холодные закуски / горячее / десерты / напитки / шоу-станции) — «линия» (Dos Toros/Chopt, П1).
- У КАЖДОЙ позиции — stepper **«порций на гостя»** (`qty` = порций на гостя, по умолч. из мин. позиций `07`) + визуализация **«покрыто N/M гостей»** (напр. канапе 3/гость → 100%). Синхрон с данными `DishCard` (`04`/`23`): `servingsPerGuest`, `station`.
- `item.total = pricePerGuest * qty * guests` (совместимо с `calcTotal` custom-режима из `07`; `qty` теперь осмысленно = порций на гостя).
- Sticky-саммери (`liveSummary`) показывает **покрытие по станциям** — клиент видит «линию» целиком.
- Мин. позиций по формату (фуршет ≥8, банкет ≥6, кофе-брейк ≥5, mobile ≥6 из `04`) соблюдается; при <50% рекоменд. → warning (edge #4).

## Сигнатурный жест «Живая линия меню» (B-MUX-2 / П6)
Критик (Запоминаемость 4/10): нет сигнатурного жеста, «вау» декларирован, не реализован. **Решение — ОДИН заданный в spec жест:**
- **«Живая линия меню»** — fullscreen-оверлей каталога меню с **parallax-каруселью станций** (каждая станция = слой с `data-speed`, Framer `useScroll`+`useTransform`, ±30px, как `GalleryTeaser`) + **on-hover dish preview**: при наведении на блюдо — крупное фото/видео во весь экран справа в реальном времени (паттерн Restaurant GEM / Awwwards SOTD).
- **Page-transition видео** между `/menu` ↔ `/plan`: Framer `AnimatePresence` + blur/clip-path морфинг (наследует clip-path wipe `09`#30). «Перелив» меню в конструктор, не «прыжок».
- **Hero-меню на главной** (опц.): полноэкранный превью хитов (≈20–30 блюд `DishCard`) как вход в `/menu` — останавливает скролл.

## Диет-бейджи и per-dish исключение аллергена (B-MUX-4 / П3)
- `04` `DishCard`/`MenuCatalog` несут `dietBadges: ('vegan'|'gluten-free'|'halal')[]` (прокинул исполнитель меню) — бейджи видимы НА карточке (не только 14 аллергенов).

- Глобальный `AllergenFilter` (`07`) и per-dish exclude (`08`) работают вместе: глобальный скрывает, per-dish — тонкая настройка внутри выбранного.

## Upgrade-модули как «вау»-предложения (B-MUX-8 / П4)

## Edge cases (критично для «не потерять клиента»)
1. guests < MIN_GUESTS[format] (10–19 для фуршета/банкета) → warning «Мин. N чел для формата X» (валидация через `07` §MIN_GUESTS). Для 10–19 фуршета/банкета — инлайн-нотис с предложением кофе-брейка/Chef-at-home (туник 10–19 закрыт, как в 07).
2. guests >500 → ветка **«Оставить заявку»** + CTA → `/api/contact` (единая терминология с 07/04, B8). slider НЕ принимает >500.
3. guests >2000 → НЕТ отдельного состояния (поглощено веткой >500, см. #2).
4. Custom <50% рекоменд. → warning.
5. 0 безалкогольных напитков → suggestion «Добавить напитки?».
6. Дата в прошлом → ошибка.
7. Дата <3 дней → «Минимальный срок 3 дня».
8. Декабрь → info «Высокий сезон, бронируйте за 2 недели».
9. Свадьба + декабрь → «Даты заполняются быстро».
10. **Chef-at-home (B-MUX-6, НЕ тупик):** при выборе `chef-at-home` переходим в почасовую ветку (Расчёт) — конструктор сразу показывает **базовую оценку «от 2 500 ₽/час (уточняется)» + пример итога** от `guestCount`×`hours`×`hourlyRateBase`, пометка «точная цена после брифа». Кнопка «Свяжитесь с менеджером» → `/api/contact` **передаёт контекст** (формат/гостей/дата/выбранные add-ons из State), НЕ «позвоните» без данных. Тупик устранён.
