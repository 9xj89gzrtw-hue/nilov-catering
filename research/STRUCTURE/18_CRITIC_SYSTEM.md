# 18 — CRITIC SYSTEM (система независимой самокритики сайта)

**Датировано:** 2026-07-18. **Источники:** реальный ресёрч 2026 (Awwwards Evaluation System,
LLM-as-Judge research 2024–2026 incl. ACL 2026 Multi-Agent-as-Judge, Google Core Web
Vitals 2026, dougvos Website Scorecard, Meta 2026 rubric-refinement).
**Зачем:** автор (я) склонен ЗАВЫШАТЬ оценку своей работы (self-serving bias /
**preference leakage** — ICLR 2026: когда генератор и оценщик = один LLM, оценка
недостоверна). Система критиков нужна, чтобы НЕ ставить себе высокие баллы незаслуженно.

---

## 1. ПРИНЦИП (из research, не из головы)

- **Rubric должен быть explicit, criterion-separated, calibrated** (Meta 2026, Adnan Masood).
  → Оценка только по заранее заданным критериям с порогами. Никаких «мне кажется круто».
- **Независимый критик ≠ автор.** Для финальной оценки используется ОТДЕЛЬНЫЙ агент
  (желательно другая модель), который НЕ генерировал код (`16`/`17` не читал как «свой»).
  → Блокирует preference leakage.
- **Объективные метрики = якорь.** Core Web Vitals (LCP/INP/CLS) и a11y-аудит —
  измеримы инструментом, не мнением. Если LCP > 2.5s → балл за Performance ≤ 3/10,
  что бы ни сказал критик-текст.
- **Multi-agent debate** (ACL 2026): critic утверждает недостатки, defender отвечает,
  judge выносит вердикт под общим rubric. Делает оценку аудируемой.

---

## 2. РОЛИ (кто оценивает)

| Роль | Кто | Задача | Может ли оценивать свой код? | Интернет? |
|---|---|---|---|---|
| **Author** | я (генератор блоков) | пишет код/контент | ❌ НЕТ — только исправляет по bug-list | — |
| **Critic (блочный)** | отд. субагент (др. модель) | ищет НЕДОСТАТКИ по rubric, занижает | ✅ только критика | ❌ (только спец-доки) |
| **FactChecker** | отд. субагент + web | сверяет claim'ы с РЕАЛЬНОСТЬЮ (гуглит цены/награды/существование) | ❌ | ✅ ОБЯЗАТЕЛЬНО |
| **Architect** | отд. субагент + web | аудит СТРУКТУРЫ: логика, связность, полнота, навигация | ❌ | ✅ (бенчмарки/конкуренты) |
| **Judge** | ещё один независимый агент/человек | сводит critic+defender, ставит балл | ✅ финальный вердикт | — |
| **Metrics** | Lighthouse/PageSpeed/axe | объективные цифры | — (якорь, не мнение) | — |
| **CodeRealityChecker** | отд. субагент + чтение репо | сверяет СПЕКУ с РЕАЛЬНЫМ КОДОМ репозитория (app/, components/, globals.css): токены/темы/цвета/наличие блоков/API-роуты | ❌ | ❌ (читает код, не веб) |

→ **Author НИКОГДА не пишет «итоговый балл».** Только Critic/FactChecker/Architect/Judge/CodeRealityChecker.
→ **FactChecker и Architect ОБЯЗАНЫ использовать интернет** (web_search/web_extract) —
  иначе они проверяют только мои же утверждения, а не истину.
→ **CodeRealityChecker ОБЯЗАН читать код репозитория** (grep app/, components/, globals.css)
  перед оценкой спек — иначе оценивает «галлюцинацию», а не реализацию. Любой рассинхрон
  спека↔код (напр. тёмная тема в коде при светлой ДНК) = БЛОКИРУЮЩИЙ баг, который
  другие критики (читающие только спецы) НЕ увидят.
→ **ПРАВИЛО ДЛЯ ВСЕХ КРИТИКОВ:** перед оценкой спек — пробежаться по реальному коду
  (хотя бы grep по `app/` и `components/`), чтобы не оценивать спеку, которой нет в коде.

## 2.1. РОЛИ СТРУКТУРНЫХ КРИТИКОВ (аудит целостности сайта)

Запускаются ПОСЛЕ того, как блочные критики довели спеки до REWORK. Проверяют не
отдельный блок, а **сайт как систему**: логична ли структура, всё ли взаимосвязано,
ничего ли не потеряно, нет ли дублей/противоречий между разделами.

| Роль | Фокус проверки | Инструмент |
|---|---|---|
| **InformationArchitect** | навигация, путь клиента (заказчика), иерархия страниц, дубли страниц, сиротские разделы | web_search (как делают конкуренты/топы) |
| **LogicAuditor** | внутренняя логика: совпадают ли цены/диапазоны/формулы между блоками, нет ли тупиков в воронке | цитаты + web (проверка claim'ов) |
| **GapHunter** | чего НЕТ, но должно быть (типичные страницы/блоки кейтеринг-сайта, которые мы упустили) | web_search (catering site essentials 2026) |
| **CoherenceChecker** | единый язык/тон/ДНК по всему сайту, нет ли разнобоя в стиле/терминах | цитаты `01`/`06` + web (best practice) |
| **FactChecker** | сверяет КАЖДЫЙ claim (19 лет, 3000+, цены, награды) с реальностью через поиск | web_search/web_extract (обязательно) |

Каждый структурный критик выставляет балл 0–10 по СВОЕМУ фокусу; итог структуры =
среднее по 5 ролям. Цель: довести до ≥8.0 через итерации (Author правит по их bug-list).

## 3. RUBRIC (8 осей, как Awwwards Evaluation System, адаптировано)

Каждая ось: **0–10**, с явными порогами. Итог = взвешенная сумма.

| # | Ось | Вес | Что проверяем | Порог «отлично» (9–10) |
|---|---|---|---|---|
| 1 | **Design / Visual** | 15% | светлая ДНК (`01`), единый стиль, negative space, типографика Cormorant (НЕ Playfair — см. `01` §3, `06` §2, `44`) | нет «винегрета», воздушно, дорого |
| 2 | **Usability / UX** | 15% | навигация, CTA-путь, mobile, нет трения | заявка за ≤3 клика, sticky CTA работает |
| 3 | **Content** | 10% | копирайт, оффер, соответствие меню/ценам | без «ЛУЧШИЙ КЕЙТЕРИНГ!!!», конкретика |
| 4 | **Technology / Perf** | 15% | **LCP<2.5s, INP<200ms, CLS<0.1** (CWV 2026), Lighthouse | все 3 метрики в «Good» |
| 5 | **Animation / Motion** | 10% | first-load wipe (`16`#30), Ken Burns, magnetic, reduced-motion | «вау при заходе», но LCP не бьёт |
| 6 | **Responsive** | 10% | mobile/tablet/desktop, без поломок | идентично качественно на всех |
| 7 | **Accessibility** | 10% | axe audit, контраст, клавиатура, alt | 0 критич. ошибок, WCAG AA |
| 8 | **SEO / Structure** | 5% | JSON-LD (`16`#27), llms.txt, meta, semantic | все схемы на месте, индексируется |
| 9 | **Benchmark gap** | 10% | покрыты ли топы из `16` (first-load, sticky CTA +31%) | все ключевые фишки топов реализованы |

> Веса и пороги — **фиксированные**, не меняются автором post-hoc.

## 4. ПРОЦЕСС ОЦЕНКИ (после каждой фазы сборки)

1. **Author** собирает блок/фазу → НЕ оценивает.
2. **Metrics:** запуск Lighthouse/PageSpeed/axe → цифры в таблицу (якорь).
3. **Critic** (субагент, др. модель) читает ТОЛЬКО спец (rubric axis 1–9) + смотрит
   деплой → пишет **bug-list** (что сломано/слабо) + ставит балл по каждой оси.
   Critic настроен НА ЗАНИЖЕНИЕ (devil's advocate).
4. **Defender** (можно = Author, но только аргументы, не балл) отвечает на bug-list.
5. **Judge** сводит → итоговый балл + вердикт «PASS / REWORK / FAIL».
6. **Author** исправляет только то, что в bug-list (не «улучшает наугад»).

> **3.5. Code-Reality Audit (ОБЯЗАТЕЛЕН перед финальным прогоном).** Отдельный
> `CodeRealityChecker` сверяет спеку с реальным кодом репозитория: grep `app/`,
> `components/`, `globals.css` на токены/темы/цвета/наличие блоков/API-роуты.
> Любой рассинхрон (напр. тёмная `#0A0A0A` в коде при светлой ДНК `01`) = БЛОКИРУЮЩИЙ
> баг. Это предотвращает оценку «галлюцинации» (спек, которой нет в коде) — что
> случилось с DeviceCritic (код тёмный, спека светлая, рассинхрон никто не заметил
> до него). Запускать CodeRealityChecker ВСЕГДА, когда код уже существует.

**Вердикты:**
- **PASS** — итог ≥ 8.0 И все объективные метрики в пороге И нет критич. a11y-ошибок.
- **REWORK** — 6.0–7.9 ИЛИ 1 ось < 5.
- **FAIL** — < 6.0 ИЛИ CWV не в «Good» ИЛИ критич. a11y-ошибка.

## 5. ЗАЩИТА ОТ SELF-BIAS (обязательные правила)

- ❌ Автор не пишет итоговый балл. Никогда. **Это относится к ЛЮБОЙ оценке, которую делает агент:**
  блочной, структурной, **и критике самих документов/структуры сайта** (напр. `40_BUILDDOC_CRITIC`,
  `42_NAV_UX_AUDIT`). Автор может составить список критиков и описать найденные баги, но
  **финальную числовую оценку (X/10) и вердикт ставит ТОЛЬКО независимый субагент** (delegate_task,
  желательно другая модель — см. SOUL «INDEPENDENT SELF-ASSESSMENT RULE»). Пишет ли агент
  «средний балл 9.0/10» на свою же работу = НАРУШЕНИЕ.
- ❌ Нельзя ставить 9–10, если объективная метрика (CWV/a11y/реальный код) не в пороге.
- ❌ Нельзя менять веса/пороги rubric после начала оценки.
- ✅ Critic — другая модель, не читавшая промежуточный код/драфт как «свой».
- ✅ Каждый балл ≥ 8 требует **доказательства** (скриншот/цифра/ссылка/цитата кода), иначе снижается до 7.
- ✅ Если Critic нашёл ≥ 3 критич. бага → итог автоматически REWORK, что бы ни сказал Judge.
- 🚀 **PROACTIVE DELEGATION (см. SOUL «PROACTIVE DELEGATION RULE»):** при любой задаче, где
  субагенты повысят эффективность (оценка качества/юзабилити/ясности, параллельный ресёрч/аудит,
  тяжёлые контексты, большие многошаговые правки), агент ОБЯЗАН использовать delegate_task вместо
  попытки сделать всё инлайн. Батч ≤3 параллельных субагента; если нужно больше — волнами.

## 6. ЧТО КРИТИК РЕАЛЬНО ПРОВЕРИТ (чек-лист для субагента)

- [ ] Hero НЕ тёмный, светлый/воздушный (`01` DNA) — или это осознанный компромисс?
- [ ] H1 использует clip-path wipe, а не просто fade (`16`#30)?
- [ ] Sticky mobile CTA есть и НЕ перекрывает контент (+31% конверсии, Contentsquare)?
- [ ] LCP < 2.5s (Poster мгновенно, видео не блокирует paint)?
- [ ] Все фото/видео заменены на новые (старых плохих НЕТ, `17`)?
- [ ] Menu/DishCard: фото + 5с loop + аллергены (14) + КБЖУ?
- [ ] TrustProof: 152-ФЗ, ТР ТС 021/2011, страховка — есть?
- [ ] JSON-LD на всех страницах (`16`#27) + llms.txt?
- [ ] reduced-motion отключает анимации?
- [ ] Навигация: заявка за ≤3 клика?
- [ ] Mobile: нет горизонтального скролла, шрифты читаемы?
- [ ] **Code-Reality:** спека совпадает с кодом репо (grep `app/`,`components/`,`globals.css`)?
  Тёмная тема в коде при светлой ДНК — БЛОКИРУЮЩИЙ баг (было с DeviceCritic).

→ Каждый ❌ = балл по оси −1, доказанный ✅ = балл держится.

---

## 7. ИТОГ

Система критиков = **rubric (9 осей, фикс. веса) + независимый Critic-агент
(др. модель) + объективные метрики (CWV/a11y) как якорь + multi-agent debate
(critic/defender/judge)**. Автор НЕ оценивает себя. Балл ≥ 8 только при доказательствах
и чистых метриках. Это блокирует preference leakage (ICLR 2026) и self-serving bias.

См. связанные: `16_BLOCK_BENCHMARKS` (цель для Critic), `17_MEDIA_DIRECTION` (чек медиа),
`01_VISUAL_DNA` (ДНК для оси Design), `06_TOKENS` (motion tokens для оси Animation).
