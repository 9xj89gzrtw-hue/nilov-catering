## 2. СПИСОК НЕСТЫКОВОК (по доказательствам)

### A. Термины / URL — СЛОМАННЫЕ КОНВЕНЦИИ (критично)
- **A1. Три словаря slug для одного типа события:**
  - `02_IA`/`03_JOURNEYS`: `/events/weddings` (plural), `/events/corporate`, `/events/private`.
  - `04_BLOCKS` блок 4 (EventTypeSelector binding, строка 154): `/events/wedding` (singular) · `/events/corporate` · `/events/private` · `/events/chef-at-home`.
  - `04_BLOCKS` блок 14 (EventHero, строки 684–687): `/events/svadba` · `/events/korporativ` · `/events/chastnoe`.
  - Плюс в том же `04`: TrustBar → `/events/wedding#hermitage` (154/122), BlogEditorial → `/events/svadba` (462), SeasonalModule → `/events/corporate` (484).
  - **Результат:** «Свадьба» адресуется 3 способами (`weddings` / `wedding` / `svadba`). Половина внутренних ссылок ведёт на несуществующие роуты. Уже подтверждено в `17_IA_CRITIC` (строка 43, «половина ссылок битая») и `21_LOGIC_AUDIT` (B5).
- **A2. Singular vs plural同一ого:** `04` блок 4 пишет «Свадьба→`/events/wedding`» (ед.), а sitemap `02:15` — `/events/weddings` (мн.). Один тип — два роута.

### B. Термины / ТАРИФЫ — модель ≠ UI (критично)
- **B1. Tier-ютion `economy|standard|premium|luxury` (08 строка 19/334, 04 строка 308) конфликтует с UI-лейблами:**
  - `04` PackageGrid (711–713): Банкет = **Классик / Премиум / VIP** (3 тарифа, нет «Люкс»). Кофе-брейк = **Лёгкий / Стандарт / Премиум / Люкс**.
  - Значит для банкета `tier==='luxury'` никогда не совпадёт с лейблом «VIP», а `tier==='classic'` вообще нет в юнионе. **Data-model и верстка говорят на разных языках** → бейдж «Популярный/recommended» и `aria-label` не сработают для банкета.
- **B2. Цена «Люкс» расходится:** Constructor (04:308) `luxury 6500`; канон/PackageGrid/07 — **Люкс фуршет = 5 350**. Один лейбл, две цифры (разница 1 150 ₽). Подтверждено `21_LOGIC_AUDIT` КР-1.
- **B3. Легаси-имена:** в `04:868` явно сказано «НЕ Classic/Premium/Imperial/Royal» — но банкет в том же файле зовётся «Классик/VIP». Остаток старой номенклатуры живёт внутри «исправленного» блока.

### C. ТОКЕНЫ — дрейф палитры и типографики
- **C1. Hex-дрейф между `01` и `06` для ОДНОЙ роли:**
  - Soft cream: `01` `#F2ECE3` → `06` `--color-secondary`/`--color-muted` `#F2EDE4` (на 1 байт).
  - Warm grey: `01` `#6B625A` → `06` `--color-muted-foreground` `#6B635B` (на 1 байт).
  - Две «истины» для одного цвета → при сборке Author возьмёт одну, дизайн поплывёт на 1hex (невидимо глазу, но ломает «single source of truth»).
- **C2. Sage (`#7E8B6B`, «в сезоне/eco») из ДНК НЕ стала токеном** — в `06` только `--color-success: #4F7A52`. Eco-метки «в сезоне» не имеют токена → будут либо хардкодом (нарушение правила `06` «НЕТ хардкод-цветов»), либо исчезнут.
- **C3. Две несовпадающие шкалы типографики:**
  - `01` §3: H1 `clamp(2.5rem,5vw,4.5rem)` (~72px max), H2 `clamp(1.8rem,3.5vw,3rem)`, H3 `clamp(1.4rem,2.5vw,2rem)`, модуль 1.333.
  - `06` §2: `--text-3xl: 44px` (h2), `--text-4xl: 59px` (h1 подстраниц), `--text-hero: 76px`.
  - px-токены `06` НЕ выводятся из rem-clamp `01` (72 vs 59/76). Авторы получают два конкурирующих «истина» для размеров заголовков — классический провал когерентности типографики (CommonPeople 2024: «headers differ in size FAQ vs Homepage»).
- **C4. Hero-размеры разъехались внутри `04`:** Home Hero (блок 1, 101) H1 `clamp(2.5rem,5vw,4.5rem)`; EventHero (блок 14, 700) H1 `clamp(44→76px)`. Две разные «единицы измерения» (rem vs px) и два разных max (72 vs 76px) для по сути одинакового Hero.

### D. MOTION — «живое фото» реализовано 3–4 способами
- `01` §4 обещает: «Ken Burns + лёгкий parallax на ВСЕХ карточках, где возможно — loop-видео на hover».
- `04` блок 5 (FormatShowcase): **Ken Burns**, loop-видео только опц. desktop.
- `04` блок 17 (DishCard): **Ken Burns по умолч., видео ПО КЛИКУ** (только хиты).
- `04` блок 18 (ShowCooking): **hover mini-video** на 6 станциях.
- `16` #2 (benchmark): «добавить короткое loop-видео на hover … у нас будет „живая еда“ как фишка».
- **Итог:** на разных секциях карточка ведёт себя визуально по-разному (зумит / играет видео при наведении / требует клика). ДНК-обещание «все фото живые» исполнено неравномерно → разнобой в визуальном языке между разделами.

### E. TONE-OF-VOICE — дрейф главного CTA-глагола
- Hero primary: **«Выбрать формат»** (→/events).
- CTASection primary: **«Спланировать событие»** (→/plan/calculator).
- StickyMobileCTA / Header: **«Спланировать»**.
- Три разных глагола для одной «главной цели». Когерентная система выбирает ОДИН первичный глагол и переиспользует. «Выбрать формат» — другой интент, чем «Спланировать».

### F. ДОКУМЕНТАЦИЯ-МУСОР (снижает доверие к контракту)
- **F1.** `01` §4 строка 62: «autoplay-видео loop + **GSAP timeline**» — хотя `01:60`, `06:125`, `05_BUILD`, `15_REPO_AUDIT` единогласно: **GSAP НЕТ в deps, только Framer**. Оговорка внутри ДНК введёт Author'а в заблуждение.
- **F2.** `06_TOKENS` §4 дублирует буллет reduced-motion дважды (строки 146–148).
- **F3.** Множественные имена одного блока: `AwardsStrip` (04 блок 3) vs `TrustBar` (04 блок 2) vs `SocialProofBar(→TrustBar)` (04:551) vs `TrustProof` (04 блок 57) vs `LiveInstagramFeed` vs `AvailabilityCalendar` — **пять+ «доверительных» блоков** с близким тоном. Риск визуального «наложения» proof-секций, что бьёт по ДНК «воздух/лёгкость» (слишком плотно).

- **Frontify (2026):** визуальная когерентность в использовании logo / color / palette across channels.
- **IxDF (2026):** независимый аудит сверяет визуальное представление (лого/цвет/типографика) на соответствие бренд-ценностям.
