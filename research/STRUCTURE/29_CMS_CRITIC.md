## ⚠️ ГЛАВНЫЙ ВЫВОД (одной строкой)

Спецификация описывает **красивый сайт, но не описывает ни одного способа, как заказчик будет его править**. ## ЧТО Я НАШЁЛ В СПЕЦАХ (цитаты — улики)

- **Цены = код.** `07_CALCULATOR_SPEC` §«Цены (канон)»: `PRICE_PER_GUEST` — хардкод-объект в `lib/pricing.ts`. `calcTotal` — чистая TS-функция без API. Поменять цену «на 100 ₽» = редактировать и пересобирать репозиторий. ❌
- **Меню = JSON.** `04_BLOCKS` (блок Constructor): «Ассеты: `catalog.json` (блюда+tier+pricePerGuest)». 78 SKU правятся ручной правкой файла. ❌
- **Видео = ручной Vimeo-id.** `04_BLOCKS` `HomeVideoShowcase` / `EventsRecap`: `clips=[{vimeoId, posterSrc, eventType, title}]` — массив прописывается в коде/данных разработчиком. Ни слова про то, КАК заказчик добавит новый рекап. ❌
- **Отзывы = массив `Review[]`** с полем `approved: boolean` (модерация «в коде»). Кто ставит `approved`? Где UI? Не описано. ❌
- **Контент-моделей в CMS нет вообще.** Все `interface` в `04` — это контракты компонентов, а НЕ схемы админки. Нет ролей, нет редактора, нет страницы «войти в админку».
- **Тексты секций** (`HeroSection.overline/h1/sub` и т.д.) в `04` помечены «из CMS/i18n, не хардкод» — НО сама CMS не определена. Обещание без механизма. ⚠️

### Смягчающие факторы (честно)
- Структура контента **чистая и регулярная**: `Review`, `Dish` (station, dietBadges, allergens, servingsPerGuest), `BlogPost`, `TeamMember`, `TestimonialsCarousel` item — всё ложится в collection-based CMS без костылей.
- `seasonalConfig`, `MIN_GUESTS`, `ADDONS` вынесены как «данные, не хардкод в JSX» — правильный рефлекс, его осталось только перенести в CMS-документ.

## РЕКОМЕНДАЦИЯ CMS (по web-исследованию 2026)

Для **нетехнического заказчика (кейтеринг, РФ, нужно править меню/отзывы/видео самому)**:

- **Storyblok** — лучший visual editor 2026 (редактор видит блоки как на сайте, live preview). Модель «блок = компонент» идеально ложится на `04_BLOCKS` (у нас страницы собираются ИЗ блоков). Минус: цена grows со слотами; для 78-SKU меню нужен отдельный подход (см. риск ниже).
- **Sanity** — гибче, дешевле при росте, отличный Studio (редактор-ориентированный), reference-поля (`Dish → Station`, `Review → EventType`). Минус: визуальный редактор слабее Storyblok, «чистый» Studio страшноват для нетехнаря без кастома.
- **Cosmic / Hygraph** — проходят мимо: Cosmic — «AI-native, dev-simplicity», но нет сильного визуального редактора для клиента; Hygraph — federation/графы, оверкилл для одного сайта.
- **Strapi/Payload (self-host)** — только если заказчик НЕ будет править сам (нужен девелопер для админки). Отметаем под цель «без девелопера».

**Вердикт критика:** **Storyblok** как основная админка (визуал + блочная сборка страниц = ровно наша `04_BLOCKS`), цены/формула — в отдельном «lockable» документе (роль «редактор» без прав на тарифы). Альтернатива — **Sanity**, если бюджет/масштаб важнее визуала.

## ТОП-РИСКИ (п.3 запроса), расшифровка

- **R-A (цены в коде vs CMS) — КРИТИЧЕСКИЙ.** `PRICE_PER_GUEST` и `ADDONS` в `lib/pricing.ts`. Любая сезонная/конкурентная правка цены требует девелопера + деплоя. Для кейтеринга, где цены меняют раз в месяц, — неприемлемо. → B-CMS-2.
- **R-B (формулы нельзя править клиентом) — ПО ФАКТУ ВЕРНО, И ЭТО ХОРОШО.** `calcTotal` (gamma-скидка, early-booking) — это бизнес-логика, её НЕЛЬЗЯ отдавать нетехнарю (сломает математику КП). Решение: клиент правит ТОЛЬКО входные (`pricePerGuest`, `addon.price`) через CMS, формула остаётся кодом. Явно зафиксировать границу.
- **R-C (Vimeo-id ручной ввод) — ВЫСОКИЙ.** Заказчик не должен копировать `vimeoId` из URL. В админке — поле «ссылка на Vimeo», парсер режет id. Иначе гарантирован битый эмбед. → B-CMS-4.
- **R-D (78 SKU в визуальном редакторе) — СРЕДНИЙ.** Storyblok visual editor тяжёл для каталога из 78 позиций; лучше `Dish` как отдельная collection (таблица/список), а не «блок на странице». Смешивать каталог с page-builder — боль. Разнести.
- **R-E (allergens пусты) — переносится из `23` R1.** Даже в CMS 14 аллергенов для 78 блюд останутся `pending`, пока заказчик не заполнит реальные значения. CMS не решит «пустые данные» — нужен этап миграции карточек заказчика в CMS. Предусмотреть bulk-импорт.

## ИТОГ ДЛЯ ПОЛЬЗОВАТЕЛЯ
«Заказчик меняет всё без девелопера» сейчас **не реализуемо (0/10)**, потому что CMS-слой в проекте **не существует**. Спецификации описывают, ЧТО показывать, но не описывают, ГДЕ и КАК лежат данные и кто ими рулит. Чтобы закрыть вопрос: выбрать **Storyblok** (визуал под нетехнаря) + вынести цены/меню/видео/отзывы в CMS-коллекции (B-CMS-1…7) + завести роль «редактор» с запретом трогать формулу (B-CMS-8). Тогда удобство правки поднимется до **8–9/10** при умеренной доработке, т.к. структура контента уже чистая.
