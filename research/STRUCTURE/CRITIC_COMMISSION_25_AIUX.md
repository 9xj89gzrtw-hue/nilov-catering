## 4. Bug-list (минимум 10; формат: file:line + альтернатива)

> Каждый баг — это конкретная строка спеки + что делать. Я НЕ «соглашаюсь», что это приемлемо.

**B-AIUX-1 · `04_BLOCKS.md:201` — имя `AIConcierge` при отсутствии AI = вводящая в заблуждение прозрачность.**
FAB назван «AIConcierge», но это «optional-обёртка над /plan/helper» (04:202) — правило, не модель.
→ **Альтернатива:** переименовать в `HelperFab`/`AssistantFab`; НЕ использовать префикс «AI», пока нет реальной модели. Если AI планируется — вынести отдельный документ со спецификацией модели/провайдера/границ.

**B-AIUX-2 · `08_CONSTRUCTOR_SPEC.md:136` vs `14_GAP_ANALYSIS.md:17` — прямое противоречие статуса AI.**
08:136: «AIConcierge **НЕ** „в резерве“» (шлёт FAB в MVP). 14:17: «AI-консьерж/чат-бот **в резерве** (не забыто, а отложено)». Два каноничных документа противоречат друг другу.
→ **Альтернатива:** один source-of-truth для статуса AI. Либо FAB убирается из MVP (следуя 14), либо 14 правится. Дублировать нельзя.

**B-AIUX-3 · `08_CONSTRUCTOR_SPEC.md:102` — «НЕ AI-консьерж, он в резерве» внутри документа, который его же и шлёт.**
Та же строка 102 говорит upgrade-подсказки — «НЕ AI-консьерж, он в резерве», а 136 тем же документом объявляет AIConcierge в MVP. Самодекларация «мы НЕ используем AI» при этом же называет FAB «AIConcierge».
→ **Альтернатива:** убрать термин «AI-консьерж» из всех строк MVP; заменить на «подборщик/помощник (rule-based)».

**B-AIUX-4 · `04_BLOCKS.md:937` / `08_CONSTRUCTOR_SPEC.md:48` — «рекомендация формата» без указания ОСНОВАНИЯ (hallucination-risk + прозрачность).**
Рекомендация = ветки `MIN_GUESTS` (10–19→кофе-брейк, 20–49→фуршет, 50+→банкет). Клиент не видит, ПОЧЕМУ предложен формат; позиционируется как «умная/персональная».
→ **Альтернатива:** инлайн-пояснение «почему: 50+ гостей → банкет, стоя → фуршет»; называть «подбор по размеру и поводу», а не «персональная рекомендация». Добавить обработку «не знаю» (бюджет 04:935) → показывать 2 варианта, не молчать.

**B-AIUX-5 · `04_BLOCKS.md:925–927` — wizard выдаётся за замену AI-консьержа без дисклеймера «это не ИИ».**
«HelpersWizard заменяет „резервный“ AIConcierge FAB» (926); клиент, видевший имя «AIConcierge», ожидает диалог, а получает дерево вопросов.
→ **Альтернатива:** в UI-копирайте wizard явно «Подборчик по 3 вопросам (не ИИ, работает по правилам)»; в спецификации зафиксировать, что это rule-based, не LLM.

**B-AIUX-6 · `04_BLOCKS.md:201` — нет a11y-спецификации FAB/помощника (доступность).**
Дан только `z-[90]` и «прячется при StickyMobileCTA». Нет `aria-label`, `aria-expanded` для панели, focus-trap, клавиатурного триггера, `prefers-reduced-motion` для самого FAB.
→ **Альтернатива:** добавить в 04_B4/B5-блок: `role="button"`, `aria-label="Помощник по подбору формата"`, открытие по Enter/Space, focus-trap панели, reduced-motion=без анимации.

**B-AIUX-7 · `04_BLOCKS.md:201` — не заданы правила НАВЯЗЧИВОСТИ FAB.**
«Прячется при StickyMobileCTA» — единственное ограничение. Не указано: показывать ли на всех роутах, авто-открывать ли при загрузке, сохранять ли dismiss (cookie/localStorage).
→ **Альтернатива:** триггер = только `/` (Hero) и `/plan`; НИКОГДА авто-открытие; состояние «закрыто» персистентно на сессию; не дублировать с `StickyMobileCTA` (уже есть) — вероятно FAB вообще избыточен.

**B-AIUX-8 · `29_CMS_CRITIC.md` (B-CMS-1…7, R-E) × `08:48`/`04:925` — помощник/рекомендации без слоя контента = fallback-дыра.**
Wizard и подсказки читают данные (форматы, `MIN_GUESTS`, `ADDONS`). 29 доказывает: CMS-слой не существует, `catalog.json`/`ADDONS` пусты (R-E). Нигде нет состояния «данные недоступны».
→ **Альтернатива:** описать error/empty-state помощника (заглушка «временно недоступно, позвоните» + tel:); guard на пустой источник данных перед рендером рекомендации.

**B-AIUX-9 · `04_BLOCKS.md:937` / `08:73–74` — нет валидации/граничных случаев рекомендации (hallucination-risk).**
Логика молча рекомендует формат. Если `guestCount` вне диапазона, `MIN_GUESTS` не определён для промежутка, или повод конфликтует с размером — поведение не специфицировано.
→ **Альтернатива:** задать явные else-ветки и приоритет (повод vs размер при конфликте); показывать «не уверены — вот 2 варианта» вместо единственной «уверенной» рекомендации.

**B-AIUX-10 · `08_CONSTRUCTOR_SPEC.md:2105–2107` × `04:1907–1909` — 152-ФЗ покрывает формы, НО не помощника/чат (приватность AI).**
`/privacy` и бейдж описывают консент для `ContactForm`/`NewsletterSignup`. Если `AIConcierge` (даже rule-based, но собирающий повод/гостей/бюджет) когда-либо станет LLM — PII пойдут на сторону без DPA/консента.
→ **Альтернатива:** зафиксировать, что wizard НЕ отправляет PII (данные остаются в браузере, `onRecommend` локален — 04:942); если появится LLM — отдельный раздел 152-ФЗ + согласие + провайдер в РФ-юрисдикции.

**B-AIUX-11 · `04_BLOCKS.md:926` / `08:136` vs `08:47` — терминологическая каша: `AIConcierge` ≠ `HelpersWizard` ≠ «шаг 1a».**
Один концепт назван тремя именами без маппинга: FAB `AIConcierge` (04:201), блок `HelpersWizard` (04:925), шаг `1a «Я не знаю, что выбрать»` (08:47). Реализатор не поймёт, что это одно и то же.
→ **Альтернатива:** единое каноничное имя компонента + таблица алиасов; убрать `AIConcierge` из имён.

**B-AIUX-12 · `04_BLOCKS.md:2244` — overclaim «персональный» при нулевой персонализации (тон/прозрачность).**
«Премиум для 50 гостей: персональный шеф, выделенный менеджер, без сноба» — сервис реально персональный, НО привязка к «AI-персональным рекомендациям» (имя консьержа) создаёт ложную связь «ИИ подобрал лично для вас».
→ **Альтернатива:** отвязать копирайт «персональный» от AI; писать «подобрали по вашему поводу и числу гостей» (факт), а не «ИИ персонально рекомендует» (вымысел).

**B-AIUX-13 · `08_CONSTRUCTOR_SPEC.md:98–102` — upgrade-подсказки («вау-эффект на глазах») не имеют триггеров доступности/данных.**
Эмоциональные подсказки (`+800 ₽/гость — вау-эффект`) тянутся из `ADDONS` (07). При пустом `ADDONS` (29 R-E) — пустота; при `prefers-reduced-motion` — не указано, гасить ли «вау»-анимацию.
→ **Альтернатива:** guard на пустой `ADDONS`; при reduced-motion — текстовая подсказка без анимации; не называть «AI-подсказкой».

## 6. Вердикт (строго ≤9)

**Итоговый скоринг AI-компонентов в структуре: 3/10.**

**Почему не выше (строго):**
1. **AI как такового нет**, а имя `AIConcierge` живёт в спецификации и на FAB — это прозрачностный провал (2/10) и потенциально обманный премиум-брендинг.
2. **Прямое противоречие между документами** (14:17 «в резерве» ↔ 08:136 «НЕ в резерве») — спецификация внутренне не согласована по статусу AI.
3. **Ноль fallback/error-состояний** для помощника и рекомендаций при пустом/битом слое контента (29 доказывает, что слой пуст).
4. **Ноль a11y-спецификации** FAB/помощника (только z-index).
5. **Overclaim «персональный/рекомендация»** при детерминированном правиле — риск hallucination-восприятия у клиента.

**Почему не 0–1 (честно, без занижения ради галочки):**
- Решение **отложить реальный AI в резерв** (14:17) — здравое продуктовое решение; MVP кейтеринга не нуждается в LLM-консьеже для конверсии.
- Детерминированный wizard `/plan/helper` (04:925–938) — **хорошая, честная замена** AI-консьержу для новичка; лучше многих «AI-чатов».
- 152-ФЗ/приватность для форм проработаны (08:2105–2107) — база под будущий LLM есть.

**Главная рекомендация критика:** **вычеркнуть термин «AI» из MVP целиком.** Либо (а) убрать FAB `AIConcierge` из MVP и оставить только честный rule-based wizard `/plan/helper` без префикса «AI»; либо (б) если AI реально планируется — написать отдельный документ `XX_AI_CONCIERGE_SPEC` с моделью, провайдером, промптами, границами, fallback, a11y и 152-ФЗ-консентом. Текущее «полу-AI» (имя без механизма) — худший из вариантов: обещает то, чего не делает.

**Что закрыть в первую очередь (blocking для премиум-репутации):** B-AIUX-2 (противоречие резерв/MVP), B-AIUX-1 (имя AI), B-AIUX-8 (fallback при пустом CMS), B-AIUX-6 (a11y FAB).
