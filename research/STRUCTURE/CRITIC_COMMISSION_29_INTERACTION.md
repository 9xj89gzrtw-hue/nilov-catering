## 3. Bug-list (14 багов, `file:line` + альтернатива)

> Формат: **[ID] Заголовок** — что сломано (цитата/строка) → **альтернатива**.

**B-INT-01 · count-up перезапускается на каждом тике слайдера (никогда не «оседает»)**
`07:190-191` описывает custom Framer-`GuestsSlider`; `07:220` «Любое изменение → price countUp»; `08:55-63` live-итог count-up «при каждом изменении». При перетаскивании ползунка (10→500, step 5 = до ~98 промежуточных значений) анимация count-up **перезапускается на каждом событии** → число «дёргается», никогда не доходит до финала, INP страдает.
→ **Альтернатива:** во время active-drag показывать точное число **без анимации** (прямой биндинг значения); count-up запускать **только на commit/release** (`onChange` завершён) либо дебаунс 250–400мс. То же для цифрового поля гостей.

**B-INT-02 · `aria-live` на цене НЕ дебаунсирован → наводнение скринридера**
`19:255` (A4): «`ResultDisplay` и `LiveSummaryBar` обернуть в `aria-live="polite"` (+`aria-atomic`) — смена цены при движении слайдера озвучивается SR». Обёртка стоит на **быстро меняющемся** значении; при drag SR прочитает все ~98 промежуточных значений подряд. A4 сам признаёт «финальное значение анонсируется без гонки анимации», но механизм дебаунса **не описан**.
→ **Альтернатива:** визуальный count-up = декоративный оверлей; для SR — отдельный `aria-live` узел с **дебаунсом 500–700мс** (анонсируется только финальное значение после паузы). Либо `aria-live` только на `ResultDisplay` (не на каждый тик слайдера).

**B-INT-03 · `GuestsSlider` как custom Framer противоречит обязательному A1 (нативный range)**
`07:190-191` («custom Framer-slider … magnetic thumb») vs `19:252` (A1): «кастомный Framer-слайдер НЕДОСТУПЕН SR+клавиатуре → Заменить на нативный `<input type="range">` с `aria-valuetext` ИЛИ `role="radiogroup"`». Спек описывает имплементацию, которую сам же запрещает — **рассинхрон не закрыт**.
→ **Альтернатива:** взять нативный `<input type="range" step="5" min="10" max="500" aria-valuetext="N гостей">`; «magnetic thumb» — чисто декоративная обводка поверх нативного, без перехвата событий (не ломает клавиатуру/точность).

**B-INT-04 · 4 дискретных тарифа представлены как slider (неправильный контрол)**
`07:189` `PackageSelector // slider 4 тарифа, цена countUp`. Тарифы — **дискретный** выбор (Эконом/Стандарт/Расширенный/Премиум), slider подразумевает непрерывную шкалу → пользователь не понимает, куда «упадёт» ползунок, SR-значение `aria-valuenow` между тарифами бессмысленно.
→ **Альтернатива:** сегментированный `role="radiogroup"` из 4 кнопок (`role="radio"` + `aria-checked`), каждая = тариф, per A1 (`19:252`). Яснее и доступнее.

**B-INT-05 · Бегущая строка (TrustBar marquee) не паузится на тач**
`04:540` «hover-pause + pause-on-focus (на `:hover`/`:focus-within`)». На тач-устройстве **нет hover/focus-within** на бегущем логотипе → логотипы «улетают», прочитать/тапнуть нельзя. K2 (`04:242`) перечисляет hover→tap для карточек/цен/тултипов, но **marquee не покрыт**.
→ **Альтернатива:** пауза по `pointerdown`/`touchstart` (drag-to-pause) ИЛИ каждый логотип — отдельная ссылка (тап останавливает поток и открывает); при `prefers-reduced-motion` — статичный ряд без анимации.

**B-INT-06 · «Haptic (mobile)» — вибро-шторм на Android + тихий no-op на iOS**
`07:220` «Любое изменение → price countUp + subtle haptic (mobile)». (а) **iOS Safari не поддерживает `navigator.vibrate`** → «mobile haptic» на iPhone молча не работает (вводящее в заблуждение заявление). (б) На Android вибрация **срабатывает на каждом тике** слайдера = вибро-шторм, раздражающий и батарейный.
→ **Альтернатива:** убрать авто-haptic на смену цены; если нужен — **один** сигнал на commit/release, `if ('vibrate' in navigator)` и под `prefers-reduced-motion: reduce` — не вибрировать.

**B-INT-07 · ContactForm: нет управления фокусом/сводкой ошибок при невалидной отправке**
`04:1015` — только «ошибки через `aria-describedby` + `role="alert"`»; `04:1064` — Zod. Не описаны: сводка ошибок (`role="alert"` блок) и **перенос фокуса на первое невалидное поле** после submit (WCAG 3.3.1, 2.4.3). Слепой/клавиатурный пользователь после отказа остаётся «в никуда».
→ **Альтернатива:** при failed submit — рендер `<div role="alert" tabindex="-1">` со списком ошибок + `focus()` на него; поля получают `aria-invalid`, `aria-describedby` на сообщение.

**B-INT-08 · Псевдо-«🔴 В эфире» живой бейдж без источника реального события**
`04:425-427` «микро-подпись живого события на Hero … капшен „реальное событие · площадка · тип“ … „🔴 В эфире: корпоративный завтрак · Петербург · 15 гостей“». Нигде нет real-time источника события → бейдж **постоянно горит** статичной демо-копией = **обманчивая индикация лайвности** (доверие/потенциально рекламная недостоверность).
→ **Альтернатива:** убрать постоянный «В эфире»; либо драйвить из реального CMS-поля текущего события (актуально/нет), либо переименовать в честное «Недавнее событие · …» без красной точки.

**B-INT-09 · AnnouncementBar: неоднозначная область dismiss (глобальный ключ скрывает все будущие анонсы)**
`04:113` «кнопка закрытия сохраняет состояние в `localStorage` (не прыгает при reload)». ZH1 (`04:114`) добавляет `per-visit`/`locked` для дедлайнов. Риск: если ключ dismiss **глобальный** (а не по `id` анонса) — пользователь, закрывший один раз, никогда не увидит ни один будущий анонс; если `per-visit` — на каждом заходе навязчиво всплывает.
→ **Альтернатива:** ключ dismissed = **`announcementId`**, а не глобальный флаг; явный `dismissScope` per item; ограничить частоту повторного показа.

**B-INT-10 · Mega-menu на hover: двойной триггер на гибридных pointer (Surface/тач-ноут)**
`04:120` (BB1): desktop mega-menu открывается на **`hover` И `click`/`Enter`/`focus`**, focus-trap внутри, Escape. На гибридных устройствах (`pointer:coarse` но hover-capable, напр. Surface/тач-ноут) `:hover` + tap могут **двойным срабатыванием** дёргать меню; K2 (`04:242`) гард `@media (hover:hover)` не применён к mega-menu.
→ **Альтернатива:** открытие **только** по click/focus/Enter (`:hover` — необязательный enhancement, обёрнутый в `@media (hover:hover) and (pointer:fine)`); focus-trap строго возвращает фокус и не ловит Tab за пределы меню.

**B-INT-11 · DishCard: 14 аллергенов скрыты за hover на desktop (безопасность — за hover)**
`04:1676-1677` «при наведении/фокусе карточки раскрываются КБЖУ + чипы 14 аллергенов». S5 (`04:1727`) делает их **всегда видимыми на mobile**, но на desktop остаётся hover-гейт. Мышь-пользователь, не наведший курсор, не видит аллергены/глютен-вердикт — критично для мамы-аллергика (Света) и Нины, даже на десктопе.
→ **Альтернатива:** критичные индикаторы (глютен-вердикт, crossContact, «подходит/не подходит») — **всегда видимы** и на desktop (как на mobile); hover/focus/tap раскрывает только детальное КБЖУ.

**B-INT-12 · `CustomMenuBuilder`: drag-drop как первичное, клавиатура только через fallback**
`04:1015` «drag-drop в custom-режиме дублируется stepper `+ / −` и `Add`-кнопкой (тач/a11y fallback)». Формулировка делает **drag-drop первичным** паттерном сборки «линии» из 78 блюд. Drag-drop на тач неточен для большой сетки; клавиатурный путь описан лишь как «fallback».
→ **Альтернатива:** сделать **stepper `+ / −` и `Add` первичным** (полностью клавиатурно-операбельным) путём; drag-drop — опциональный enhancement, не несущий функциональности, недоступной иначе.

**B-INT-13 · Переходы между шагами (blur/clip-path) не гарантированы под reduced-motion кодом**
`08:87` «Page-transition видео … Framer `AnimatePresence` + blur/clip-path морфинг»; `08:141` «Переходы между шагами: Framer Motion `layout` + AnimatePresence». `08:140-144` и `04:1013` говорят reduced-motion «выключает layout-морфинг», но это **гайдлайн, не механизм**: нет вызова `useReducedMotion()` вокруг `AnimatePresence`, blur/clip могут проиграть вестибулярно-чувствительному юзеру.
→ **Альтернатива:** обернуть каждый `AnimatePresence`/`motion` в `useReducedMotion()` → при `reduce` мгновенный swap (opacity 0/1 без blur/clip/translate), без морфинга.

**B-INT-14 · Превью «X–Y ₽» на шаге гостей создаёт ложную неопределённость**
`08:49` (шаг 2) «preview „X–Y ₽“». Расчёт `calcTotal` **детерминирован**, но UI показывает **диапазон** вместо точного live-итога → выглядит как «инструмент не может посчитать» / искусственная неопределённость (вредит доверию Марины/Игоря).
→ **Альтернатива:** показывать **точный** live-итог (тот же `liveSummary`), а не разброс; диапазон уместен только если реально неизвестен формат/тариф.

## 5. Вердикт

**Итоговый скоринг взаимодействий: 6.4 / 10** (строго, ≤9).

**Что работает:** связный каркас мастера/калькулятора с running-итогом; продуманный mobile/tablet-контракт (ровно 1 нижний sticky, 44px, safe-area, Lenis off на тач); контракт hover→tap (K2/K3/K6); `focus-visible` gold; честные `pending-verification`-состояния календаря/аллергенов; строгая верификация внешнего рейтинга.

**Что ломает взаимодействие (top-5 по тяжести):**
1. **B-INT-03** — custom Framer-слайдер вопреки обязательному A1 (a11y-рассинхрон spec↔critic).
2. **B-INT-02** — `aria-live` на цене без дебаунса (наводнение SR).
3. **B-INT-01** — count-up перезапускается на каждом тике (число не оседает, INP).
4. **B-INT-06** — vibrate-шторм на Android / no-op на iOS.
5. **B-INT-11** — аллергены скрыты за hover на desktop (безопасность).

**Условие PASS (≥8):** закрыть B-INT-01…14, в первую очередь привести имплементацию слайдеров/тарифов к A1 (нативный range + radiogroup), дебаунсить `aria-live`, убрать авто-haptic, вынести критичные аллергены из-под hover на всех pointer, и закрыть управление фокусом/сводкой ошибок в формах. До этого — **не готово к премиум-уровню взаимодействия**.
